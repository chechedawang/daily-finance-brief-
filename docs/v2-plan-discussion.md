## 问题一：新闻数量多 + 重复 → 质量筛选 Top 20

核心需求是引入**质量打分 + 去重**，然后截取 top 20。

### 关于「点击量」的困境

RSS 源**不提供点击量/阅读量数据**，只能拿到标题、链接、摘要、时间。因此需要用**代理信号（proxy signals）**来近似衡量热度：

| 信号 | 含义 | 可行性 |
|---|---|---|
| **跨源覆盖度** | 同一事件被多家报道 → 说明是热点 | ✅ 完全可控，纯算法 |
| **时间新鲜度** | 发布时间越接近当前，权重越高 | ✅ 已有时间数据 |
| **来源权威度** | 头部媒体的报道权重更高（可配置） | ✅ 主观但合理 |
| **内容充实度** | 有摘要、标题信息量大的文章优先 | ✅ 从现有字段可得 |
| **关键词密度** | 含公司名/股票代码/政策关键词的文章加分 | ✅ 可维护关键词表 |

### 去重思路

用标题相似度做聚类（去掉特殊符号后归一化比较，编辑距离或余弦相似度均可）。同一聚类只保留得分最高的那条。

### 打分公式草案

```
score = 跨源覆盖加成 + 新鲜度分 + 源权重 + 内容丰富度 + 关键词加分
```

按 score 降序取 top 20。

> 会不会「误杀」好内容？有可能，但从 135+ 条筛到 20 条，少量漏网是值得的。后续根据实际效果调整权重或关键词表即可。

## 问题二：从手动刷新 → 每日中午 12:00 自动刷新

契合「每日快报」定位，但受限于 **Render 免费层 15 分钟无请求休眠**。

### 时间窗口：前一天 12:00 ~ 当天 12:00

- 覆盖 A 股前一天下午收盘后的消息
- 覆盖当天早间新闻（政策发布、美股收盘等）
- 在午间给出一个「上午 + 昨夜」的完整热点简报

> ⚠️ 需要在筛排时**只保留时间落在窗口内的文章**，RSS 返回的可能是近几天内容。

### 方案 A：外部定时唤醒 + 应用层判断（推荐）

```
┌─────────────────┐    每天 12:00 CST     ┌──────────────────┐
│ 免费 cron 服务   │ ──── ping ──────────→ │ Render 服务       │
│ (cron-job.org)  │     /api/refresh       │ 被唤醒 → 抓取 →   │
│                  │                        │ 筛 Top20 → 写缓存 │
└─────────────────┘                        └──────────────────┘
                                                    │
                                          用户访问时直接读缓存
```

- 用免费的 cron-job.org 或 GitHub Actions 每天 12:00（北京时间）ping 一次 `/api/refresh`
- 同时，用户访问时检查「当前缓存数据是否属于今天的 12:00 批次」，如果不是且当前时间已过 12:00，自动触发一次刷新
- 双重保障确保数据新鲜

### 方案 B：纯应用层懒刷新

不做外部 ping，只在用户请求时判断：如果缓存为空、或缓存日期不是「今天」且当前时间 > 12:00，就触发刷新。

> 缺点：每天第一个用户在 12:00 后访问会经历冷启动（约 20s 抓取 + 筛排）。

### 方案 C：保留手动刷新作为补充

即使主要靠每天 12:00 自动刷新，保留刷新按钮/下拉刷新作为兜底，成本几乎为零。

## 问题三：代码架构规划

当前全堆一个文件在雏形阶段没问题，但后续加新闻源、加打分策略、加用户偏好、甚至加数据库持久化时，单文件就是灾难。

### 建议的分层架构

```
project/
├── app.py                  # Flask 应用工厂 + 路由注册（薄薄一层）
├── config.py               # 全局配置：SOURCES 列表、超时、时间窗口等
│
├── fetchers/               # 抓取层 —— 只负责「拿数据」
│   ├── __init__.py          #   注册表：RSS_FETCHERS / API_FETCHERS
│   ├── base.py              #   基类 / 公共逻辑
│   ├── rss.py               #   feedparser 统一抓取
│   └── wallstreetcn.py      #   华尔街见闻等 API 源专用逻辑
│
├── services/               # 业务层 —— 「怎么处理数据」
│   ├── __init__.py
│   ├── orchestrator.py      #   并发调度（ThreadPoolExecutor 那套）
│   ├── dedup.py             #   去重（标题相似度聚类）
│   └── ranker.py            #   打分排序（跨源覆盖 + 新鲜度 + 权重）
│
├── cache.py                 # 缓存层（现在简单，以后可换 Redis）
│
├── templates/
│   └── index.html
│
├── static/                  # 待拆分：独立的 CSS + JS 文件
│   ├── style.css
│   └── app.js
│
├── requirements.txt
└── CLAUDE.md
```

### 核心原则

- **fetchers/** 只管「从外部拿原始数据」，每次加新源只需新增一个文件 + 在注册表里挂载
- **services/** 管「数据怎么处理和筛选」，与数据来源无关
- **app.py** 管路由胶水层，越薄越好
- **config.py** 管所有可调的参数（源列表、超时、top N 数量、打分权重等），改参数不碰逻辑代码

---

## 建议实施顺序

1. **先做架构拆分** —— 在单文件状态做重构最简单，拆完再做功能不容易出 bug
2. **再做去重 + 打分 + Top 20** —— 这是核心价值，直接影响用户体验
3. **最后做定时刷新** —— 依赖外部 cron，需要额外配置，但改动量最小
4. **刷新按钮暂时保留** —— 不碍事，作为定时刷新的补充
